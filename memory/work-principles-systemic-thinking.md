# 工作原则：系统性思维

## 核心要求

做任何工作都需要有系统性思维，但**剂量要匹配任务复杂度**。简单任务靠直觉，复杂任务必须显式建立系统。

## 关键要素

1. **循环而非线性**
   - 不是「分析 → 执行 → 结束」，而是「定义 → 执行 → 观察偏差 → 修正规则 → 再执行」。
   - 每次小修正都是系统演进的一部分。

2. **先分层，再分块**
   - 层：长期原则、项目规范、单次操作。
   - 块：同一层内拆成可独立处理的部分。
   - 混乱通常源于层与块混在一起。

3. **把隐性约定变成显性规则**
   - 心里知道不够，要写成文件、加入索引、放进读取规则表。
   - 例如：新项目默认落 `E:\projects\`、memory 指针只存跨项目引用。

4. **警惕过度系统**
   - 系统是为了减少决策负担，不是为了增加。
   - 定期清理：哪些规则已失效、哪些约定重复、哪些可以合并。

5. **反馈比规划重要**
   - 再漂亮的系统也要能接收真实反馈。
   - 例如：`import_md.py` 同步数据库、SQL 查询验证关系网络、git commit 做安全网。

## 当前场景中的应用

| 场景 | 系统性思维体现 |
|---|---|
| 心理学知识库 | Markdown 是唯一真相源，所有工具围绕它转 |
| 卡片填充工作流 | 先建网，再填卡——顺序比速度重要 |
| Memory 管理 | 双轨记忆系统——项目约定跟项目走，个人记忆跨项目复用 |
| 磁盘/工作区 | Claude 配置在 E 盘，新项目默认落 E 盘——原则一致，入口不同 |

## Why

没有系统性思维，复杂工作会快速退化为碎片和重复劳动；过度系统又会拖累执行。目标是**刚好够用的结构**。

## How to apply

- 接手任何非琐碎任务时，先问：目标是什么？边界在哪里？有哪些重复模式？
- 做完一次后，把可复用的部分写成规则或模板。
- 定期回顾已有规则，删除冗余，合并重复。
