## 两套记忆系统的关系

我体内天生带一个自动记忆系统（`claude-mem`，MCP Server 版），你又亲手建了一套文件式档案（Markdown）。它们不是竞争关系，是**双轨互补**。

| 维度 | 自带 `claude-mem` | 自建文件 Memory |
|------|------------------|----------------|
| **定位** | 短期/对话记忆（海马体） | 长期/个人知识库（百科全书） |
| **存储** | 数据库，自动捕获 | Markdown，手动整理 |
| **人能不能直接看** | ❌ 只能通过查询接口 | ✅ 打开就能读 |
| **擅长记** | 碎片、临时决策、对话上下文 | 稳定事实、项目档案、资源清单 |
| **维护** | 被动积累 | 主动整理 |
| **注入方式** | 第二次会话起自动注入相关上下文 | 每次会话自动注入 `MEMORY.md` 索引 |

## 什么东西该往哪儿放

### ✅ 放自建档案（Markdown）

- **用户画像**：你是谁、学什么、沟通偏好
- **项目档案**：项目路径、技术栈、功能清单、设计决策
- **资源清单**：GitHub 仓库、工具链接、论文集合
- **经验教训**：踩过的坑、验证过的原则、工作流约定
- **环境配置**：盘符规划、junction、npm 前缀、技能列表
- **模式/子模式入口**：只放触发方式和指向项目内部规范的路径，不重复项目资料内容

> 原则：**不会随对话变化的事实性知识**，整理后入库。
> **项目内部规范（如 spec-*.md、模式约定、子模式规则）应放在项目内部**，由 memory 中的入口文件指向，不要记在 memory 里。避免项目资料与跨项目个人记忆混淆，也避免内容重复后不同步。

### ✅ 放自带系统（自动记忆）

- 某次对话里提到的临时想法
- "上次我们讨论到一半的方案"
- 临时的偏好调整（还没确定要不要长期执行）
- 对话中的决策过程、争论、反转

> 原则：**动态、碎片、临时**的信息，让系统自动吃进去。

## 理想工作流

```
对话中产生信息
    ↓
自带系统自动捕获（事件 + 观察）
    ↓
我定期复盘 → 提取精华 → 写入 Markdown 档案
    ↓
下次对话 = MEMORY.md 索引 + claude-mem 相关上下文 同时注入
```

- **2026-06-11**：误以为 claude-mem 完全不可用（skill 版装失败），后来发现 MCP 版天生自带，只是当前项目数据库为空。
- **教训**：不要把「skill 安装失败」等同于「功能不存在」，先检查工具列表。

## 记忆入口化规则

如果某类记忆只在特定情境下才需要，就不要让它每次会话都注入。采用「入口文件 + 按需读取」：

| 记忆类型 | 入口文件位置 | 详细资料位置 | 读取时机 |
|---|---|---|---|
| 模式专属记忆 | `memory/` 下的轻量入口 | **项目内部**的模式资料目录（如 `meta/`、`00-prompts/`） | 用户说出触发指令时 |
| 项目专属记忆 | `memory/projects-index.md` | `memory/projects/{项目名}/` | 用户明确提到该项目时 |

规则：
1. 入口文件只放触发方式/路径，不写详细内容。
2. 详细资料互相隔离，读到 A 项目时不要主动读 B 项目。
3. 不要主动联想，等用户提到再读取。
